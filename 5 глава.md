# 5 РАЗРАБОТКА ФУНКЦИОНАЛЬНОЙ ЧАСТИ ПРОГРАММЫ

## 5.1 Алгоритм переноса одного файла в корзину

Функциональная обработка «одного удаления» сосредоточена в модуле `trash_hook.c` и общих функциях `trash_common.c`. Такой подход позволяет не смешивать перехват интерфейса libc с размещением данных на диске и записью метаданных. На вход логики переноса попадает уже выбранный прикладной программой путь (через `unlink()`, `unlinkat()` без `AT_REMOVEDIR` или `remove()` для не-каталога).

Алгоритм переноса одного объекта включает следующие этапы:

– получение абсолютного пути (`resolve_absolute_from_dirfd`: абсолютный аргумент, либо текущий каталог при `AT_FDCWD`, либо путь каталога по `dirfd` через `/proc/self/fd/…`);

– проверка условий, при которых перенос в корзину не выполняется (`TRASH_HOOK_DISABLE`, рекурсивный вход `in_hook`, для `unlinkat` — флаг `AT_REMOVEDIR`, путь вне `$HOME`, неподходящий тип объекта и т.п.); в этих случаях — вызов реальных `unlink`/`unlinkat` из libc;

– `lstat` объекта: при ошибке — проброс поведения libc; если объект не обычный файл или не находится в дереве `$HOME` — реальное удаление без корзины;

– подготовка структуры корзины (`trash_ensure_layout`), вычисление путей `Trash/files` и `Trash/info`;

– генерация уникального имени записи и целевого пути в `files`;

– заполнение структуры `trash_entry` (канонический исходный путь, права, владелец, временные метки, время удаления);

– перенос содержимого: преимущественно `rename()` в каталог `files`; при `EXDEV` — послойное копирование и затем реальный `unlink` исходного имени;

– запись файла метаданных `*.trashinfo` в `info`; при ошибке записи — удаление частично созданного файла в `files` и возврат ошибки.

Следовательно, алгоритм переноса одного файла образует законченный локальный конвейер: он может выполняться при каждом вызове удаления внутри процесса с подключённым `libtrashhook.so`, без изменения исходного кода приложения. Автономность этого конвейера на уровне одного вызова позволяет многократно обрабатывать удаления, инициируемые разными программами и пользовательскими действиями.

## 5.2 Алгоритм работы утилиты управления корзиной

В проекте отсутствует пул процессов и пакетная диспетчеризация; функциональную «оркестрацию» после накопления данных в корзине выполняет утилита `trash_util.c`. Её можно рассматривать как второй функциональный контур: работа уже не с перехватом libc, а с каталогами `files` и `info`.

Алгоритм по режимам можно представить в следующем виде.

**Режим `list`:** открыть каталог `info`; для каждого имени с суффиксом `.trashinfo` разобрать файл (`trash_parse_info`); вывести идентификатор записи, дату удаления и исходный путь.

**Режим `restore <ID>`:** сформировать пути к объекту в `files` и к `.trashinfo`; проверить наличие файла данных; при необходимости создать недостающие родительские каталоги для исходного пути; при конфликте имени на месте восстановления — получить уникальное имя (`trash_resolve_unique_path`) и выполнить `rename`; восстановить режим доступа, владельца и времена (`chmod`/`chown`/`utimensat`); удалить метафайл.

**Режим `purge`:** рекурсивно удалить содержимое `files` и `info` (обход `nftw`), затем пересоздать раскладку корзины (`trash_ensure_layout`).

**Режим `info`:** подсчитать число `.trashinfo` и суммарный размер каталога `files`.

**Режим `menu`:** интерактивный выбор перечисленных операций с подтверждением полной очистки.

Пример логической последовательности: удаление файла при активном перехватчике, просмотр списка записей в корзине, восстановление выбранной записи — приведён на рисунке 5.2.

## 5.3 Форматы входных и выходных данных

На уровне системной программы входные и выходные данные описываются структурно. Перехватчик не интерпретирует содержимое файлов как документы или медиа; он оперирует путями и метаданными файловой системы. Утилита оперирует уже сформированными записями корзины.

Структура основных сущностей приведена в таблице 5.1.

**Таблица 5.1** – Форматы и назначение данных в проекте

| Объект | Расположение | Назначение |
|--------|--------------|------------|
| Исходные обычные файлы пользователя | Любой путь внутри `$HOME`, например два файла в одном подкаталоге домашней директории | Объекты, при удалении с подключённым `libtrashhook.so` подлежащие переносу в корзину (при выполнении условий алгоритма). |
| Тело удалённого файла | `~/.local/share/Trash/files/<id>` | Сохранённое содержимое под уникальным именем записи. |
| Метаданные записи | `~/.local/share/Trash/info/<id>.trashinfo` | Исходный путь, время удаления, права, владелец, временные метки для восстановления. |
| Переменная `LD_PRELOAD` | Окружение процесса, выполняющего удаление | Подключение `libtrashhook.so` для перехвата `unlink`/`unlinkat`/`remove`. |
| Переменная `HOME` | Окружение процесса | Граница дерева, внутри которого включён перенос в корзину. |
| Переменная `TRASH_HOOK_DISABLE` | Окружение процесса | Отключение перехвата при ненулевом значении (кроме строки `0`). |

Для проверки работы комплекса достаточно подготовить один или несколько обычных файлов в каталоге внутри `$HOME` и выполнить удаление с подключением библиотеки перехвата; содержательный тип данных при этом не ограничивается.

Для проверки функциональности выполнялись позитивные и негативные испытания: сборка проекта, удаление с `LD_PRELOAD` и без него, работа утилиты в режимах командной строки и меню, проверка нескольких записей в корзине, восстановление с конфликтом имён, очистка и статистика.

Результаты контрольных испытаний приведены в таблице 5.2.

**Таблица 5.2** – Результаты контрольных испытаний

| Проверка | Ожидаемый результат | Фактический результат |
|----------|---------------------|------------------------|
| Сборка проекта<br>командой `make` | В каталоге сборки появляются `libtrashhook.so` и `trash-util`; процесс компиляции и линковки завершается без ошибок, исполняемые права на утилиту позволяют её запуск. | Проверка пройдена успешно: оба целевых файла созданы, в выводе `make` отсутствуют сообщения об аварийном завершении компиляции. |
| Удаление тестового файла в `$HOME` командой<br>`LD_PRELOAD="$PWD/libtrashhook.so" rm путь` | Исчезновение имени по исходному пути; появление тела файла в `~/.local/share/Trash/files` и метафайла в `~/.local/share/Trash/info` с расширением `.trashinfo`. | Проверка пройдена успешно: `ls` по исходному пути даёт отказ «No such file», в каталогах корзины отображаются новый файл данных и соответствующий `.trashinfo`. |

| Удаление пути вне дерева `$HOME`<br>(например, под `/tmp`) с тем же `LD_PRELOAD` | Немедленное удаление без переноса; каталоги `Trash/files` и `Trash/info` не получают новой пары «данные + мета» для этого пути. | Проверка пройдена успешно: файл отсутствует по исходному пути, количество записей в корзине не увеличилось. |

| Запуск `./trash-util info` | Число файлов в корзине и суммарный размер в байтах согласованы с фактическим содержимым каталога `files`. | Проверка пройдена успешно: после известного числа удалений показатель количества совпадает с ожиданием, размер ненулевой при наличии данных. |

Полученные результаты подтверждают, что функциональная часть комплекса корректно выполняет перенос в корзину при подключённом перехватчике, обслуживание записей через `trash-util` и предсказуемое поведение при отключении перехвата и для путей вне домашнего каталога.
