### 7.1 Сборка программы

Для сборки проекта необходимо перейти в корневой каталог и выполнить команду:

`make`

В результате формируются два артефакта:

- `libtrashhook.so` — динамическая библиотека перехвата удаления;
- `trash-util` — консольная утилита управления корзиной.

Для удаления собранных файлов и временных объектных артефактов используется команда:

`make clean`

Для штатного использования проекта в системе должны быть доступны инструменты, приведенные в таблице 7.1.

| Компонент | Назначение |
|---|---|
| `gcc` | Компиляция исходного кода на языке C. |
| `make` | Автоматизация сборки и очистки проекта. |
| `bash` | Запуск команд с `LD_PRELOAD` и интерактивная работа с утилитой. |
| `ld.so`/подсистема динамической загрузки | Подключение `libtrashhook.so` через переменную `LD_PRELOAD`. |

### 7.2 Формат запуска

Проект используется в двух режимах:

1. запуск внешней команды удаления с подключенным перехватчиком;
2. запуск утилиты управления корзиной.

Общий формат мягкого удаления через перехват:

`LD_PRELOAD="<путь_к>/libtrashhook.so" <команда_удаления> <файл>`

Общий формат запуска утилиты:

`./trash-util <команда> [параметры]`

Назначение основных команд приведено в таблице 7.2.

| Параметр/команда | Назначение |
|---|---|
| `list` | Показать содержимое корзины (ID, дата удаления, исходный путь). |
| `restore <ID>` | Восстановить файл из корзины по идентификатору. |
| `info` | Вывести статистику: число файлов и занятый объем. |
| `purge` | Полностью очистить корзину. |
| `menu` | Запустить интерактивный режим меню. |
| `TRASH_HOOK_DISABLE=1` | Временно отключить перехват удаления для команды. |
| `-h` (не используется) | Отдельный ключ справки не реализован; справка выводится при неизвестной команде. |

### 7.3 Примеры использования

1. Сборка проекта:

`make`

2. Мягкое удаление файла из домашнего каталога:

`LD_PRELOAD="$PWD/libtrashhook.so" rm "$HOME/test_area/file_a.txt"`

3. Просмотр содержимого корзины:

`./trash-util list`

4. Восстановление файла по ID:

`./trash-util restore 1713191111_123456789_1234567890`

5. Получение статистики корзины:

`./trash-util info`

6. Очистка корзины:

`./trash-util purge`

7. Запуск интерактивного меню:

`./trash-util menu`

После запуска утилита выводит диагностические сообщения о результате операций. Типичные сообщения включают:

- «Восстановлено: ...»;
- «Конфликт: восстановлено как ...»;
- «Корзина пуста.»;
- «Корзина очищена.».

### 7.4 Возможные ошибки и рекомендации по устранению

| Сообщение/ситуация | Возможная причина | Рекомендация |
|---|---|---|
| `./trash-util: No such file or directory` | Проект не собран или запуск выполняется не из каталога проекта. | Выполнить `make` и проверить текущий каталог. |
| `ERROR: ld.so: object '.../libtrashhook.so' from LD_PRELOAD cannot be preloaded` | Неверный путь к библиотеке или отсутствует файл `libtrashhook.so`. | Проверить путь и собрать проект повторно: `make`. |
| Файл удалился окончательно и не появился в корзине | Файл находился вне `$HOME` или запуск был без `LD_PRELOAD`. | Использовать путь в домашнем каталоге и корректный запуск с `LD_PRELOAD`. |
| `Элемент с ID '...' не найден в files.` | Неверный ID или объект уже восстановлен/удален. | Повторно выполнить `./trash-util list` и использовать актуальный ID. |
| `parse info: ...` при восстановлении | Поврежден или отсутствует `.trashinfo`. | Проверить каталог `~/.local/share/Trash/info`; при необходимости удалить неконсистентные записи вручную. |
| `Ошибка: ... Permission denied` | Недостаточно прав на файлы/каталоги корзины или целевой путь восстановления. | Проверить права доступа к `$HOME/.local/share/Trash` и целевым каталогам. |
| Конфликт имен при восстановлении | В исходном пути уже существует файл с тем же именем. | Это штатное поведение: файл будет восстановлен с суффиксом `.1`, `.2`, ... |
| Очистка корзины выполнена, нужные файлы исчезли | Выполнена необратимая команда `purge`. | Использовать `purge` только после проверки содержимого корзины через `list`. |

Таблица 7.3 - Типовые ошибки и способы их устранения

При работе с перехватчиком рекомендуется сначала использовать тестовые файлы в отдельном каталоге внутри `$HOME`, а для системных и критичных данных применять дополнительные меры защиты (резервное копирование, контрольный просмотр через `list` перед `purge`).
